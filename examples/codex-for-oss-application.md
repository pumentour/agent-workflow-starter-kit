# Codex For Open Source Application Checklist

This checklist helps maintainers prepare a stronger open-source support application.

It does not guarantee acceptance.

## Before Applying

Check whether the repository has:

- Public visibility
- Clear README
- License
- Real examples
- Maintainer activity
- Evidence of use
- Clear project purpose

## Weak Signals

A repository looks weak if it has:

- No README
- No examples
- No license
- No issues or commits
- No stars or forks
- A vague project purpose

## Stronger Signals

A repository looks stronger when it has:

- Concrete workflows
- Real-world examples
- Documentation in more than one language if relevant
- Clear maintainer role
- External traffic or community interest
- Stars, forks, issues, or contributions

## Application Fields

Prepare:

- Email connected to the account
- GitHub username
- Public repository URL
- Maintainer role
- Repository qualification statement
- OpenAI organization ID
- API credit usage plan
- Extra context

## Qualification Statement Template

```text
This repository provides practical templates for solo builders using local-first AI agent workflows. It includes AGENTS.md patterns, task cards, tool routing, verification habits, and editable asset workflows. I am the primary maintainer and use these patterns in real production work.
```

## API Credit Usage Template

```text
I will use API credits to build maintainer automation: issue triage, PR review assistance, test failure analysis, documentation updates, release note generation, and security-oriented code review. The goal is to reduce manual maintenance load while improving reliability and contributor response time.
```

## Reality Check

If the repository is new, submit only if the cost is low and expectations are modest.

For better odds, first improve:

- README depth
- Examples
- Public activity
- Stars or forks
- Clear maintainer history
